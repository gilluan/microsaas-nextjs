# Data Model: Initial MicroSaaS Platform Foundation

## Entity Definitions

### User
Primary entity representing registered users of the platform.

**Fields:**
- `id`: String (UUID) - Primary identifier, auto-generated by Cognito
- `email`: String (required) - User's email address, must be unique
- `name`: String (required) - User's display name
- `avatar`: String (optional) - S3 key for profile image
- `emailVerified`: Boolean - Email verification status
- `createdAt`: AWSDateTime - Account creation timestamp
- `updatedAt`: AWSDateTime - Last profile update timestamp
- `lastLoginAt`: AWSDateTime (optional) - Last successful login

**Validation Rules:**
- Email must be valid email format (Zod email validation)
- Name minimum 2 characters, maximum 50 characters
- Avatar must be valid S3 key if provided
- Password requirements: min 8 chars, uppercase, lowercase, number

**Authorization:**
- Owner can read/update their own profile
- Public read access for basic profile information (name, avatar) for social features
- Admin users can read all profiles

### UserActivity
Tracks user activities for the dashboard activity feed.

**Fields:**
- `id`: String (UUID) - Primary identifier
- `userId`: String (required) - Foreign key to User
- `activityType`: Enum - Type of activity (LOGIN, PROFILE_UPDATE, etc.)
- `description`: String - Human-readable activity description
- `metadata`: AWSJSON (optional) - Additional activity data
- `createdAt`: AWSDateTime - Activity timestamp

**Activity Types:**
- `LOGIN` - User logged in
- `PROFILE_UPDATE` - User updated their profile
- `AVATAR_UPLOAD` - User uploaded/changed avatar
- `PASSWORD_CHANGE` - User changed password
- `EMAIL_VERIFICATION` - User verified email

**Relationships:**
- `belongsTo` User via `userId`

**Authorization:**
- Owner can read their own activities only
- Admin users can read all activities

## GraphQL Schema Structure

### User Model
```graphql
type User @model @auth(rules: [
  { allow: owner, operations: [read, update] },
  { allow: public, operations: [read], fields: ["id", "name", "avatar"] }
]) {
  id: ID!
  email: AWSEmail! @auth(rules: [{ allow: owner, operations: [read] }])
  name: String!
  avatar: String
  emailVerified: Boolean! @default(value: false)
  activities: [UserActivity] @hasMany(indexName: "byUser", fields: ["id"])
  createdAt: AWSDateTime!
  updatedAt: AWSDateTime!
  lastLoginAt: AWSDateTime
}
```

### UserActivity Model
```graphql
enum ActivityType {
  LOGIN
  PROFILE_UPDATE
  AVATAR_UPLOAD
  PASSWORD_CHANGE
  EMAIL_VERIFICATION
}

type UserActivity @model @auth(rules: [
  { allow: owner, ownerField: "userId", operations: [create, read] },
  { allow: private, operations: [create] }
]) {
  id: ID!
  userId: ID! @index(name: "byUser", sortKeyFields: ["createdAt"])
  user: User @belongsTo(fields: ["userId"])
  activityType: ActivityType!
  description: String!
  metadata: AWSJSON
  createdAt: AWSDateTime!
}
```

## Data Relationships

### User → UserActivity (One-to-Many)
- One User can have many UserActivity records
- UserActivity.userId references User.id
- Enables activity feed queries sorted by timestamp
- Owner-based authorization ensures users only see their activities

## Data Access Patterns

### User Profile Management
```graphql
# Get user profile with basic info
query GetUserProfile($id: ID!) {
  getUser(id: $id) {
    id
    name
    email
    avatar
    emailVerified
    createdAt
    lastLoginAt
  }
}

# Update user profile
mutation UpdateUserProfile($input: UpdateUserInput!) {
  updateUser(input: $input) {
    id
    name
    email
    avatar
    updatedAt
  }
}
```

### Activity Feed
```graphql
# Get user's recent activities
query ListUserActivities($userId: ID!, $limit: Int = 10) {
  listUserActivities(
    filter: { userId: { eq: $userId } }
    limit: $limit
    sortDirection: DESC
  ) {
    items {
      id
      activityType
      description
      createdAt
      metadata
    }
  }
}

# Subscribe to real-time activity updates
subscription OnUserActivity($userId: ID!) {
  onCreateUserActivity(filter: { userId: { eq: $userId } }) {
    id
    activityType
    description
    createdAt
    metadata
  }
}
```

## Zod Validation Schemas

### User Validation
```typescript
import { z } from 'zod'

export const UserProfileSchema = z.object({
  name: z.string()
    .min(2, 'Name must be at least 2 characters')
    .max(50, 'Name must be less than 50 characters')
    .trim(),
  email: z.string()
    .email('Invalid email address')
    .toLowerCase(),
  avatar: z.string().url().optional()
})

export const PasswordSchema = z.string()
  .min(8, 'Password must be at least 8 characters')
  .regex(/[A-Z]/, 'Password must contain at least one uppercase letter')
  .regex(/[a-z]/, 'Password must contain at least one lowercase letter')
  .regex(/[0-9]/, 'Password must contain at least one number')

export const UserRegistrationSchema = UserProfileSchema.extend({
  password: PasswordSchema,
  confirmPassword: z.string()
}).refine(data => data.password === data.confirmPassword, {
  message: 'Passwords do not match',
  path: ['confirmPassword']
})
```

### Activity Validation
```typescript
export const ActivityTypeSchema = z.enum([
  'LOGIN',
  'PROFILE_UPDATE',
  'AVATAR_UPLOAD',
  'PASSWORD_CHANGE',
  'EMAIL_VERIFICATION'
])

export const UserActivitySchema = z.object({
  activityType: ActivityTypeSchema,
  description: z.string().min(1).max(200),
  metadata: z.record(z.unknown()).optional()
})
```

## State Transitions

### User Registration Flow
1. **Unregistered** → **Pending Verification**
   - User submits registration form
   - Cognito creates user account with `emailVerified: false`
   - Verification email sent

2. **Pending Verification** → **Active**
   - User clicks verification link
   - `emailVerified` set to `true`
   - LOGIN activity created

### Profile Update Flow
1. User submits profile changes
2. Zod validation on client side
3. GraphQL mutation with server-side validation
4. Profile updated in database
5. PROFILE_UPDATE activity created
6. Real-time subscription notifies dashboard

### Avatar Upload Flow
1. User selects image file
2. Client-side file validation (type, size)
3. Request pre-signed URL from S3
4. Direct upload to S3 with progress tracking
5. Update user profile with S3 key
6. AVATAR_UPLOAD activity created

## Multi-tenancy Considerations

### Data Isolation
- User data isolated by Cognito user ID (owner field)
- GraphQL authorization rules prevent cross-tenant access
- S3 file organization by user ID prefix
- Activity data scoped to individual users

### Scaling Patterns
- User table partitioned by creation date for archival
- Activity table uses GSI for efficient user-based queries
- Connection pooling managed by AppSync
- Read replicas for reporting queries (future)

## Privacy and Compliance

### Data Retention
- User profiles retained as long as account active
- Activity logs retained for 90 days (configurable)
- Deleted user data purged within 30 days

### Personal Data Handling
- Email addresses encrypted at rest
- Avatar uploads scanned for inappropriate content
- Audit trail for all profile modifications
- GDPR compliance with data export/deletion capabilities
